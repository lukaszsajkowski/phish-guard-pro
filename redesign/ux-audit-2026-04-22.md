## Critical findings

### [C-01] Password toggle button missing `aria-label`

**Component:** `frontend/src/app/page.tsx:165` and `frontend/src/app/register/page.tsx:162`
**Standard violated:** 12 — Accessibility (`aria-label` on all icon-only buttons)
**Current behavior:** The show/hide password `<button>` elements in both login and register pages have no `aria-label`. Screen readers announce them as unlabeled buttons.
**Expected behavior:** Each toggle button should have an `aria-label` like `"Show password"` / `"Hide password"` that changes with state.
**Fix:** Add `aria-label={showPassword ? "Hide password" : "Show password"}` to both password toggle buttons in `page.tsx:165` and `register/page.tsx:162`.

### [C-02] Session history detail page right panel not responsive — hidden on mobile

**Component:** `frontend/src/app/history/[sessionId]/page.tsx:386`
**Standard violated:** 3 — Information density and layout; 12 — Accessibility
**Current behavior:** The intel `<aside>` has a fixed `w-[320px]` with no responsive breakpoint. On narrow screens the panel is invisible/overflows, making IOC data inaccessible on mobile.
**Expected behavior:** On smaller screens, the intel panel should either collapse to a sheet/drawer (like the sidebar does) or stack below the chat area.
**Fix:** Add responsive classes: at `lg` breakpoint show as sidebar, below `lg` either render as a `Sheet` or stack below the chat. Mirror the collapsible side-panel pattern already used on the dashboard page.

## High findings

### [H-01] Loading states use centered spinner instead of skeleton screens

**Component:** `frontend/src/app/history/page.tsx:124`, `frontend/src/app/history/[sessionId]/page.tsx:326`, `frontend/src/app/dashboard/page.tsx:38-47`
**Standard violated:** 11 — Loading and empty states (skeleton screens instead of centered spinners)
**Current behavior:** All three pages show a `<Loader2>` spinner with "Loading..." text while fetching data. This gives the user no preview of the page structure and increases perceived wait time.
**Expected behavior:** Skeleton screens that mirror the final layout — skeleton cards for history list, skeleton header + chat area for session detail, etc. The `skeleton.tsx` UI component already exists but is unused.
**Fix:** Replace `<Loader2>` loading states with skeleton layouts using the existing `frontend/src/components/ui/skeleton.tsx` component. For the history list: render 3-5 skeleton `Card` rows. For the session detail: skeleton header bar + skeleton chat bubbles + skeleton sidebar.

### [H-02] IOC values displayed in `text-xs` (12px) throughout IntelDashboard — too small for frequently-read data

**Component:** `frontend/src/components/dashboard/IntelDashboard.tsx:322` (IOC value), lines 274, 279, 312, 448
**Standard violated:** 13 — Typography (minimum `text-sm`/14px for frequently-read data)
**Current behavior:** IOC values (`font-mono text-xs`), IOC labels (`text-xs font-medium`), threat scores, and raw enrichment data are all rendered at 12px. These are high-value data that analysts scan repeatedly.
**Expected behavior:** IOC values and labels should be at least `text-sm` (14px). Supporting metadata (source, latency) can stay at `text-xs`.
**Fix:** Change IOC value `<p>` at line 322 from `font-mono text-xs` to `font-mono text-sm`. Change IOC label at line 312 from `text-xs font-medium` to `text-sm font-medium`. Leave timeline events, raw-data preview, and metadata at `text-xs`.

### [H-03] No search or filtering on session history list

**Component:** `frontend/src/app/history/page.tsx`
**Standard violated:** 9 — Session history and management (filtering by attack type, date range, risk level; search by IOC value or persona name)
**Current behavior:** The history page shows a flat paginated list with no ability to filter or search. As sessions accumulate, finding a specific session requires paging through every page.
**Expected behavior:** At minimum: a search input (filter by persona name or attack type) and filter chips/dropdowns for attack type and risk level. Date range filter is a lower-priority addition.
**Fix:** Add a search bar and filter row above `SessionHistoryList`. Pass filter params as query parameters to the API (`?attack_type=ceo_fraud&min_risk=5&search=John`). This requires both frontend and backend work.

### [H-04] No toast/notification for successful export actions

**Component:** `frontend/src/app/dashboard/page.tsx:707-764` (handleExportJson, handleExportCsv), `frontend/src/app/history/[sessionId]/page.tsx:206-276`
**Standard violated:** 10 — Notifications and feedback (toast notifications for successful actions)
**Current behavior:** After a successful JSON/CSV export, the file simply downloads silently. If the download fails, no error is shown (the `catch` blocks are empty or only `console.error`).
**Expected behavior:** Show a toast notification ("Export complete — phishguard_session_2026-04-22.json") on success and an error toast on failure. Also show toast for clipboard copy actions in `IntelDashboard`.
**Fix:** Add a toast provider (shadcn/ui `sonner` or `toast` component) and trigger success/error toasts in all export handlers and the copy-to-clipboard callback.

### [H-05] `SessionSummary` IOC list shows raw IOC values without defanging URLs

**Component:** `frontend/src/components/dashboard/SessionSummary.tsx:148`
**Standard violated:** 6 — Threat intelligence display (defang URLs to prevent accidental clicks)
**Current behavior:** IOC values in the session summary are rendered as plain text: `<p className="font-mono text-sm truncate">{ioc.value}</p>`. URL-type IOCs display the raw URL which could be accidentally clicked or copied unsafely.
**Expected behavior:** URL IOCs should be defanged using the existing `defangUrl()` utility from `@/lib/utils`.
**Fix:** Import `defangUrl` and change line 148 to: `{ioc.ioc_type === "url" ? defangUrl(ioc.value) : ioc.value}`.

## Medium findings

### [M-01] `ClassificationResult` card uses hardcoded light-mode colors

**Component:** `frontend/src/components/app/ClassificationResult.tsx:43,54-56`
**Standard violated:** 2 — Dark mode and contrast
**Current behavior:** Uses `border-red-200 bg-red-50/10`, `bg-green-100 text-green-700`, `bg-yellow-100 text-yellow-700`, `bg-blue-100 text-blue-700` — all light-mode palette colors that don't adapt to dark mode. The confidence badge becomes hard to read on dark backgrounds.
**Expected behavior:** Use dark-mode-aware classes like `dark:bg-green-900/30 dark:text-green-400` (pattern already used in `ATTACK_TYPE_COLORS`).
**Fix:** Add dark-mode variants to the card border/bg and confidence badge classes. E.g., `border-red-200 dark:border-red-800 bg-red-50/10 dark:bg-red-900/10` and similar for the confidence badge.

### [M-02] Risk score color thresholds inconsistent between `ioc.ts` and `risk.ts`

**Component:** `frontend/src/lib/constants/ioc.ts:69-78` vs `frontend/src/lib/constants/risk.ts:100-108`
**Standard violated:** 1 — Severity color coding consistency (same thresholds everywhere)
**Current behavior:** `getRiskScoreColor` in `ioc.ts` uses 4 tiers: `>=8` red, `>=5` orange, `>=3` yellow, else green. `getRiskLevelColor` in `risk.ts` uses 3 tiers: `high` red, `medium` orange, `low` green — but the mapping from score to level is done server-side, so the boundary between "medium" and "high" may differ from the 8-point threshold used client-side.
**Expected behavior:** One source of truth for score-to-color mapping, used everywhere.
**Fix:** Consolidate: have `risk.ts` functions derive from the same thresholds used in `ioc.ts`, or better yet, move all risk-color logic into a single file. Ensure `SessionHistoryList`, `IntelDashboard`, and `RiskScoreBreakdown` all use the same utility.

### [M-03] Timeline section lacks "View all" progressive disclosure

**Component:** `frontend/src/components/dashboard/IntelDashboard.tsx:566`
**Standard violated:** 4 — Progressive disclosure (timeline limited to recent N events with "View all")
**Current behavior:** Timeline renders all events in a `max-h-48 overflow-y-auto` scrollable area. No limit on initial display, no "View all" button.
**Expected behavior:** Show the most recent 5 events with a "View all (N events)" expand button if there are more.
**Fix:** Slice timeline to the first 5 items by default, add a "View all" toggle button below if `timeline.length > 5`.

### [M-04] PersonaCard uses custom design tokens not matching rest of dashboard

**Component:** `frontend/src/components/dashboard/PersonaCard.tsx:17-18,37-41`
**Standard violated:** 2 — Dark mode and contrast; 13 — Typography
**Current behavior:** Uses `bg-surface`, `border-border2`, `text-text`, `text-text-muted`, `text-text-secondary`, `text-[17px]`, `text-[12px]`, `text-[13px]`, `text-[11px]` — custom design token classes from the redesign. Other dashboard components use standard shadcn/ui tokens (`bg-card`, `text-foreground`, `text-muted-foreground`, `text-sm`, `text-xs`).
**Expected behavior:** Consistent token usage across all components. Either all components use the new design tokens or all use shadcn/ui tokens — not a mix.
**Fix:** This is a transitional issue from the design token overhaul (US-042). Either finish migrating all dashboard components to the new tokens, or revert PersonaCard to shadcn/ui tokens for consistency.

### [M-05] Session detail header has duplicate theme toggle

**Component:** `frontend/src/components/history/SessionDetailHeader.tsx:138-155`
**Standard violated:** 3 — Information density (max 5-7 primary widgets per view)
**Current behavior:** The session detail header has its own theme toggle button using a custom SVG sun icon. The `AuthenticatedLayout` already provides global theme toggle via `ThemeToggle` component in `AppHeader`. This creates two theme toggles on the page.
**Expected behavior:** One theme toggle in a consistent location (sidebar or top-right header), not duplicated per-page.
**Fix:** Remove the `onToggleTheme` prop and theme toggle button from `SessionDetailHeader`. The global `ThemeToggle` in the authenticated layout handles this.

### [M-06] Export dropdown in `SessionDetailHeader` is custom-built instead of using shadcn/ui

**Component:** `frontend/src/components/history/SessionDetailHeader.tsx:172-221`
**Standard violated:** 12 — Accessibility (focus management, keyboard navigation)
**Current behavior:** The export dropdown is a manually-implemented dropdown with `useRef` click-outside detection. It lacks keyboard navigation (arrow keys, Escape to close), `role="menu"`, and `aria-expanded` attributes.
**Expected behavior:** Use shadcn/ui `DropdownMenu` component which handles all accessibility patterns out of the box.
**Fix:** Replace the custom dropdown with `DropdownMenu`, `DropdownMenuTrigger`, `DropdownMenuContent`, and `DropdownMenuItem` from shadcn/ui.

### [M-07] No STIX 2.1 export option

**Component:** `frontend/src/components/dashboard/SessionSummary.tsx:163-191`, `frontend/src/components/history/SessionDetailHeader.tsx:157-221`
**Standard violated:** 8 — Data export UX (multiple formats: JSON, CSV, STIX 2.1)
**Current behavior:** Only JSON and CSV export are available.
**Expected behavior:** Threat intelligence platforms expect STIX 2.1 formatted exports for interoperability with SOC/SIEM tools.
**Fix:** Add a STIX 2.1 export button alongside JSON and CSV. This requires backend support for STIX serialization, but the frontend button should be present (disabled with tooltip if not yet implemented).

### [M-08] `ReadOnlyChatArea` copy buttons lack `aria-label`

**Component:** `frontend/src/components/history/ReadOnlyChatArea.tsx:117-133`
**Standard violated:** 12 — Accessibility (`aria-label` on all icon-only buttons)
**Current behavior:** Copy buttons in the read-only chat show text "Copy to clipboard" next to the icon, but the button element itself has no `aria-label`. While the visible text helps, the button element should have an explicit label for consistency.
**Expected behavior:** Add `aria-label="Copy message to clipboard"` on the copy button element.
**Fix:** Add `aria-label="Copy message to clipboard"` to the copy `<button>` at line 118.

## Low findings

### [L-01] No entrance animation for newly extracted IOCs

**Component:** `frontend/src/components/dashboard/IntelDashboard.tsx:292-488`
**Standard violated:** 14 — Micro-interactions (entrance animations for new IOCs)
**Current behavior:** When new IOCs are extracted during a conversation, they appear instantly in the list with no transition.
**Expected behavior:** New IOC cards should fade-in/slide-in with a brief animation, and high-value IOCs should get a pulse effect.
**Fix:** Add `animate-in fade-in slide-in-from-top-2 duration-300` to IOC card divs, keyed so the animation triggers only on mount.

### [L-02] Risk score in `IntelDashboard` fallback mode lacks animated transition

**Component:** `frontend/src/components/dashboard/IntelDashboard.tsx:527-548`
**Standard violated:** 14 — Micro-interactions (animated risk score transitions)
**Current behavior:** The risk score number jumps instantly when it changes.
**Expected behavior:** Score should count up/down smoothly when it updates. The progress bar already has `transition-all duration-500` but the number does not animate.
**Fix:** Add a `useCountUp` hook or CSS transition on the score `<div>` to animate the number from old to new value.

### [L-03] Pagination could use keyboard shortcuts for power users

**Component:** `frontend/src/components/history/Pagination.tsx`
**Standard violated:** 12 — Accessibility (keyboard navigation enhancement)
**Current behavior:** Pagination buttons are individually keyboard-accessible but there are no keyboard shortcuts (e.g., left/right arrow keys for prev/next page).
**Expected behavior:** Arrow key navigation when pagination is focused.
**Fix:** Add `onKeyDown` handler on the `<nav>` element to handle ArrowLeft (prev) and ArrowRight (next) key events.

### [L-04] `ScammerInput` shows `Ctrl+Enter` hint but not `Cmd+Enter` on macOS

**Component:** `frontend/src/components/dashboard/ScammerInput.tsx:78`
**Standard violated:** 10 — Notifications and feedback (clear UX hints)
**Current behavior:** The hint always shows `Ctrl+Enter` regardless of platform.
**Expected behavior:** On macOS, show `Cmd+Enter`; on other platforms, show `Ctrl+Enter`.
**Fix:** Use `navigator.platform` or `navigator.userAgentData` to detect macOS and display the appropriate modifier key.

### [L-05] `EmailInput` card has unused `CardHeader` and `CardDescription` imports

**Component:** `frontend/src/components/app/email-input.tsx:11-12`
**Standard violated:** N/A — Code hygiene
**Current behavior:** `CardHeader`, `CardTitle`, `CardDescription` are imported but not used in the JSX.
**Expected behavior:** Clean imports.
**Fix:** Remove unused imports.

## What's done well

- **URL defanging** is properly implemented in `IntelDashboard` using `defangUrl()` — prevents accidental clicks on malicious URLs
- **IOCs grouped by type** in `IntelDashboard` with summary count badges — follows SOC dashboard standard
- **Progressive disclosure** in `RiskScoreBreakdown` with expand/collapse and proper `aria-expanded`/`aria-controls`
- **Agent thinking panel** uses `<details>` for natural progressive disclosure with no JS overhead
- **Keyboard navigation** on session history cards — proper `role="button"`, `tabIndex={0}`, and Enter/Space handlers
- **Collapsible side panel** on dashboard with responsive auto-collapse — good information density management
- **Risk score progress bars** have `role="progressbar"` with `aria-valuenow/min/max` — proper accessibility
- **`prefers-reduced-motion`** support in global CSS — animations are disabled for users who prefer it
- **Pagination** has full `aria-label` attributes on all controls and proper `aria-current="page"`
- **Mobile sidebar** uses Sheet component with proper screen reader title — good responsive pattern
- **Copy-to-clipboard** feedback with checkmark icon and "Copied!" text — good micro-interaction
- **Consistent error handling** via `ApiError` component with retry button — reused across all pages
- **Enrichment results** show threat score with visual bar, reputation badge, and expandable raw data — good progressive disclosure
- **Empty states** with helpful CTAs ("Start analyzing" button in history) — good empty state pattern
