# Product Requirements Document (PRD) - PhishGuardPro

## 1. Product Overview

PhishGuard is an autonomous agent-based system implementing the Active Defense
paradigm in the fight against phishing. Unlike traditional, passive
anti-phishing systems (which block and forget), PhishGuard actively engages
attackers in believable conversation.

The system operates in Manual Simulation mode (Sandbox), where the user pastes
the content of a suspicious email, and four specialized AI agents (orchestrated
via LangGraph) collaborate to:

- Classify the type of phishing attack
- Generate an appropriate "victim" persona
- Conduct an engaging but safe conversation
- Extract Indicators of Compromise (IOC) in real-time

The product serves as a showcase of AI architecture capabilities in the
cybersecurity domain, built on an Enterprise-grade SaaS stack.

### Business Goals

| Goal | Description |
| ---- | ----------- |
| Tarpitting (Time Waste) | Maximize the time attackers waste with the bot |
| Threat Intel Extraction | Extract key indicators (wallets, IBANs, domains) |
| Safety | Demonstrate safe use of LLM with no real user data leakage |

## 2. User Problem

### Problem Context

Security researchers and security enthusiasts face the following challenges
when analyzing phishing campaigns:

1. Lack of tools for safe interaction with scammers to collect threat intel
2. Manual conversation management is time-consuming and requires attention
3. Risk of accidentally revealing real personal data when engaging attackers
4. Difficulty in systematically documenting and extracting IOC
5. Lack of standardized methodology for classifying phishing attack types

### Target User

Security Researcher / Hobbyist - a technical person who:

- Manually analyzes suspicious phishing emails
- Wants to collect intelligence on attacker tactics and infrastructure
- Needs a safe sandbox environment for experiments
- Appreciates transparency in AI tool operation

### User Needs

| Need | PhishGuard Solution |
| ---- | ------------------- |
| Quick attack classification | Profiler Agent with 8 categories + confidence |
| Believable persona | Persona Engine with 4-6 profiles, auto-select |
| Data security | Safety Layer with dual-mode validation (input + output) |
| IOC collection | Intel Collector with real-time extraction to dashboard |
| Documentation | JSON/CSV export of conversation history and intel |

## 3. Functional Requirements

### 3.1 Input Analysis Module (Priority: P0)

FR-001: System accepts pasted phishing email content in a text field.

FR-002: Profiler Agent classifies email into one of 8 categories:

- Nigerian 419 Scam
- CEO Fraud
- Fake Invoice
- Romance Scam
- Tech Support Scam
- Lottery/Prize Scam
- Crypto Investment Scam
- Delivery Scam

FR-003: System displays confidence score (0-100%) for the selected
classification.

FR-004: System handles NOT_PHISHING classification with force-continue option
for user.

### 3.2 Persona Engine Module (Priority: P0)

FR-005: System automatically selects persona from library of 4-6 profiles:

- Naive Retiree
- Stressed Manager
- Greedy Investor
- Confused Student
- (optionally 2 additional)

FR-006: Each persona has defined attributes:

- Name (generated by Faker)
- Age
- Writing style (formality, typical errors, Caps Lock usage)
- Characteristic phrases

FR-007: FakeDataGenerator seeded per-session ensures persona data consistency.

FR-008: Agent responses are styled according to selected persona.

### 3.3 Conversation Loop Module (Priority: P0)

FR-009: System generates responses turn-by-turn using "loose goals" strategy.

FR-010: Conversation Planner defines goals to achieve in conversation:

- Obtaining payment details
- Extending conversation
- Building trust
- Asking open-ended questions

FR-011: User can edit generated response before copying (Edit before copy
mode).

FR-012: System displays "Copy to clipboard" button for ready response.

FR-013: User pastes scammer's response in dedicated field, initiating next
turn.

### 3.4 Safety Layer Module (Priority: P0)

FR-014: Safety Check validates every generated response before presentation.

FR-015: System blocks responses containing:

- Real PII formats (national ID numbers, SSN, real email addresses)
- Real corporate domains from blocklist
- Sensitive financial data formats

FR-016: Validation uses regex patterns + keyword blocklist.

FR-017: Upon detecting unsafe content, system automatically regenerates
response.

FR-018: Intel Collector in dual-mode also validates agent responses (not just
scammer messages).

### 3.5 Intel Dashboard Module (Priority: P0)

FR-019: Side panel displays 4 sections:

- Attack type + confidence score
- List of collected IOC (icons + values)
- Risk Score (scale 1-10)
- Extraction timeline

FR-020: System extracts IOC from scammer messages:

- BTC wallet addresses (pattern: bc1/1/3 prefix)
- IBAN numbers (pattern: 2 letters + 2 digits + ...)
- Phone numbers (international formats)
- Malicious URLs

FR-021: Color coding for priorities:

- Red: high value intel (wallet, IBAN)
- Yellow: partial/lower priority intel

FR-022: Dashboard updates in real-time after each scammer message.

### 3.6 Agent Thinking Module (Priority: P1)

FR-023: Collapsible "Agent Thinking" panel shows:

- Current turn goal
- Selected tactic
- Agent reasoning

FR-024: Panel is collapsed by default, expandable on user request.

### 3.7 Session Management Module (Priority: P1)

FR-025: System implements soft limit of 20 message exchanges per session.

FR-026: Upon reaching limit, system displays message with continuation option.

FR-027: "End and summarize" button generates session final report.

FR-028: System detects "end of game":

- Scammer uses phrases indicating unmasking
- No response for 2+ turns in simulation

FR-029: Upon end-of-game detection, system automatically proposes summary.

### 3.8 Export Module (Priority: P1)

FR-030: User can export conversation history to JSON.

FR-031: User can export collected IOC to CSV.

FR-032: Export includes metadata: timestamp, attack type, persona, session
metrics.

FR-033: Export buttons available in side panel and session summary.

### 3.9 Demo Mode Module (Priority: P2)

FR-034: System offers 2-3 pre-loaded demonstration scenarios.

FR-035: Scenarios can be browsed step by step without API calls.

FR-036: Demo mode accessible from application start screen.

### 3.10 Error Handling Module (Priority: P1)

FR-037: System implements retry with exponential backoff (max 3 attempts) for
API calls.

FR-038: On timeout/error, user-friendly message is displayed with "Try again"
option.

FR-039: System implements graceful degradation - fallback to cheaper model
when overloaded.

FR-040: System caches system prompts for optimization.

## 4. Product Boundaries

### In Scope

| Functionality | Status |
| ------------- | ------ |
| Manual simulation mode (Sandbox) | Yes |
| Pasting phishing email content | Yes |
| Automatic attack type classification (8 categories) | Yes |
| Automatic persona selection (4-6 profiles) | Yes |
| Turn-by-turn conversation loop (LangGraph orchestration) | Yes |
| Fake data generation (Faker) | Yes |
| Safety validation (regex + blocklist) | Yes |
| Intel Dashboard with real-time IOC extraction | Yes |
| Export to JSON/CSV | Yes |
| Next.js UI (React) | Yes |
| Demo mode with pre-loaded scenarios | Yes |
| Persistence in Supabase (Postgres) | Yes |
| User authentication (Supabase Auth) | Yes |

### Out of Scope

| Functionality | Exclusion Reason |
| ------------- | ---------------- |
| Automatic email sending/receiving (SMTP/IMAP) | Spam risk, complexity |
| Image generation (fake confirmations) | Complexity, abuse risk |
| Multilingual support | Planned for future |
| User persona customization | Planned for future |
| ML-based NER for IOC | Uses regex |
| Mobile responsiveness | Desktop-first |

### Technical Constraints

| Constraint | Implication |
| ---------- | ----------- |
| Language: English only | LLM handles English scams better |
| Supabase persistence | Data stored securely in Postgres |
| API rate limits | Soft limit 20 turns, retry logic |
| Regex-based IOC extraction | ~90% accuracy for known formats |

### US-032: Enhanced Risk Score ✅

Title: Multi-dimensional threat risk assessment

Description: As a security researcher, I want to see a detailed, multi-factor
risk score to better understand the threat level and scammer tactics.

Acceptance Criteria:

- Risk Score calculated from 6 weighted components:
  - Attack Severity (25%): CEO Fraud=4, Crypto=4, Nigerian 419=3, etc.
  - IOC Quality (25%): BTC wallet=3, IBAN=3, phone=2, URL=1
  - IOC Quantity (15%): +0.5 per IOC up to max 1.5 points
  - Scammer Engagement (15%): Based on response length and frequency
  - Urgency Tactics (10%): Detection of pressure keywords (urgent, deadline)
  - Personalization (10%): Scammer uses victim name or references context
- Risk Score displayed with breakdown panel (expandable)
- Each component shows individual score and contribution
- Color coding: Green (1-3), Yellow (4-6), Red (7-10)
- Breakdown updates in real-time as conversation progresses
- Algorithm is transparent and explainable to user
